# ingest-references パイプライン

pagegrabで収集したリファレンスファイルを一括処理し、現環境に統合する。
**目的**: 外部から収集した知見を、開発環境・まいの運用・事業戦略に反映する。

## 引数
- `--dry-run`: トリアージ+ナレッジ抽出まで実行し、Issue作成・実装はスキップ
- `--no-implement`: Issue作成まで実行し、実装はスキップ（次セッションで対応）

## 入力元（これ以外は参照しない）

```
<YOUR_DOWNLOADS>/pagegrab/text/*.md
```

**IMPORTANT**: 入力はこのディレクトリの `.md` ファイルのみ。以下は入力元ではない:
- `pagegrab/images/` -- テキストファイルに対応する画像がある場合のみ補助的に参照
- `pagegrab/analysis/` -- スキャンしない
- `research/mai-references/` 直下 -- スキャンしない（アーカイブ先であり入力元ではない）

## 処理先

```
research/mai-references/
├── INDEX.md                       ← 全ファイル台帳（SSOT）
├── archive/extracted/             ← ナレッジ化済みMDファイルの移動先
└── archive/consumed-media/        ← 処理済みPDF/PNG

~/.claude/knowledge/  ← ナレッジファイル群（統合先）
```

## 実行フロー

### Step 0: 新規ファイルの検出

1. `<YOUR_DOWNLOADS>/pagegrab/text/*.md` を全件リスト
2. `research/mai-references/INDEX.md` を読み込み、既にintegrated/discardedされたファイル名を除外
3. **未処理ファイル数が0件** → 「新規リファレンスなし」と報告して終了

**出力**: 未処理ファイル一覧（ファイル名のみ）をユーザーに表示

### Step 1: テキスト読み込み + トリアージ

`pagegrab/text/` 内の未処理 `.md` ファイルのみを読む。

対応する画像がある場合（`pagegrab/images/{同名スラグ}/`）、画像枚数を記録するが、画像の中身は必要な場合のみ読む。

**他のナレッジファイルや作業ファイルはこのステップでは読まない。**

#### トリアージ基準

| 判定 | 基準 | 処理 |
|------|------|------|
| **HIGH** | 具体的なコード/手法/数値データがある。現環境に直接適用可能 | ナレッジ抽出 + Issue作成 + 実装 |
| **MEDIUM** | 参考になる事例やパターンがある。将来使える可能性 | ナレッジ抽出のみ（Issue不要） |
| **DISCARD** | テキストが薄い/機能アナウンスのみ/知見なし/重複 | **ユーザー確認後**、INDEX.mdに記録して終了 |

#### トリアージの判断軸（優先度順）

1. **開発環境の強化**: Claude Code skill設計、自動化パターン、CI/CD、テスト手法
2. **事業・プロダクト改善**: SEO/AEO、マーケティング手法、収益化パターン、UX改善
3. **クライアントワーク**: 広告運用、レポーティング、ツール連携
4. **まいのストーリー**: AI秘書運用事例、成長物語の素材、SNS運用テクニック
5. **インフラ・セキュリティ**: OpenClaw運用、サーバー構成、セキュリティパターン

#### DISCARDの明確な基準
- テキストが見出しのみ（本文が画像に依存し、画像が保存されていない）
- 単なる機能アナウンス/リリースノート（新しい知見なし）
- INDEX.mdで既にintegrated/discardedされたURLの再キャプチャ
- 他者の個人的な話（事業/技術的な学びがない）

**CRITICAL: DISCARD前のユーザー確認**
- pagegrabでわざわざキャプチャした = ユーザーに意図がある。「テキストが薄い」は自分の読み取り能力の限界であり、ユーザーの意図とは別
- トリアージ結果テーブルを提示した後、DISCARD候補について「なぜキャプチャしたか」をユーザーに確認する
- ユーザー承認なしにファイルを削除しない
- ユーザーが意図を説明したら、その視点で再トリアージする（デザインリファレンス、文体の手本、事業アイデアの種、人格設計の参考 等）

**出力**: トリアージ結果テーブルをユーザーに表示。**DISCARD候補がある場合は削除前にユーザー確認を求める**

```
| # | ファイル名 | 判定 | 概要 | 統合先候補 |
|---|-----------|------|------|-----------|
```

### Step 2: ナレッジ抽出（HIGH + MEDIUM を並列処理）

**サブエージェントを並列起動**して、各ファイルのナレッジを抽出・統合する。

サブエージェントには以下を渡す:
- ソースファイルの内容（メインエージェントが読んだテキストをプロンプトに含める）
- 統合先ナレッジファイルのパス

**サブエージェントが読むのは統合先ナレッジファイル1つのみ。** ソースのテキストはプロンプトに含めて渡す。

#### サブエージェントへの指示テンプレート

```
## タスク: {ファイル名}からナレッジ抽出

### ソース内容
{ここにテキストを貼り付ける}

### 統合先
{統合先ナレッジファイルの絶対パス}

### やること
1. 統合先ナレッジファイルを読み込み、重複を確認
2. 新規知見を統合先ナレッジに追記（セクション追加 or 既存セクション拡充）
3. 統合先が存在しない場合は新規作成
4. 追加箇所に出典（ソース名、日付）を明記
5. 既存コンテンツは変更しない
```

#### ナレッジ統合先の選定ルール

| カテゴリ | 統合先ファイル |
|---------|--------------|
| Claude Code / Skill設計 | `claude-code-skill-patterns.md` |
| SEO / AEO | `aeo-strategy.md`, `seo-best-practices.md` |
| SNS運用（まい固有） | `social-media-operation.md` |
| SNS運用（汎用テクニック） | `social-media-strategy.md`（なければ新規） |
| OpenClaw / 自動化 | `openclaw-reference.md`, `automation-infrastructure.md` |
| 広告運用 | `google-ads-automation.md` |
| クライアントワーク | `client-{name}.md` |
| デザイン | `document-design.md` |
| セキュリティ | `web-security.md` |
| 事業戦略 | `business.md`, `decisions.md` |
| プロダクト固有 | `product-{name}.md` |
| 新カテゴリ | 新規ファイル作成（`~/.claude/knowledge/`配下） |

#### まいへの適用判断

以下のいずれかに該当する場合、`social-media-operation.md` にも統合する:
- AI秘書/AIエージェントの運用事例（まいの成長ストーリーの素材）
- SNSバズ/エンゲージメント手法（まいの投稿戦略に直接適用）
- パーソナルブランディング手法（まいのキャラクター強化に使える）
- 自動化パターン（まいのパイプライン改善に使える）
- 収益化事例（まいの将来Phase設計の参考）

### Step 3: アーカイブ + INDEX.md更新

1. 処理済みMDファイルを `pagegrab/text/` から `research/mai-references/archive/extracted/` に移動
2. DISCARDファイルは `pagegrab/text/` から削除
3. INDEX.md に新しいセクションを追記:

#### Knowledge INDEX.md 更新

新規ナレッジファイル作成 or 既存ファイルに統合した場合:
1. `~/.claude/knowledge/INDEX.md` を読む
2. 統合先ファイルが既存クラスタに含まれているか確認
3. 含まれていなければ適切なクラスタに追加
4. 新規ファイルの場合は新クラスタ作成 or 既存クラスタに追加

```markdown
## YYYY-MM-DD 追加分

| # | ファイル名 | 概要 | 統合先 | 状態 |
|---|-----------|------|--------|------|
| {次番号} | `{filename}` | {概要} | {統合先ファイル} | **integrated** |
| -- | `{filename}` | {概要} | -- | **discarded** |
```

### Step 4: Issue作成（HIGH判定のもののみ）

**`--dry-run` の場合はスキップ。**

HIGH判定のナレッジから、現環境に適用するためのアクションをIssue化する。

#### Issue作成基準
- 具体的な実装タスクがあるもの（コード変更、設定追加、スクリプト作成）
- 既存Issueと重複しないこと（`gh issue list`で確認）
- 1ナレッジ = 0~2 Issue（アクションがなければIssue不要）

#### Issue作成先の判定

| 対象 | リポジトリ |
|------|-----------|
| <YOUR_OPS_REPO>全般、まい、クライアントワーク、インフラ | `<YOUR_GITHUB_USER>/<YOUR_OPS_REPO>` |
| FCBench固有 | `<YOUR_GITHUB_USER>/my-product-a-repo` |
| BizBench固有 | `<YOUR_GITHUB_USER>/my-product-b` |
| 偉人Vlogメーカー固有 | `<YOUR_GITHUB_USER>/my-product-c` |

#### Issue本文テンプレート
```markdown
## 背景
{ナレッジの要約 + 出典}
**ナレッジ**: `{統合先ナレッジファイルパス}`

## やること
{具体的なタスクリスト}

## 関連
{関連する既存Issue番号}
```

### Step 5: 実装（サブエージェント並列）

**`--dry-run` または `--no-implement` の場合はスキップ。**

Issue作成後、サブエージェントを起動して実装する。

- 独立したIssueは並列サブエージェントで同時実装
- 依存関係があるIssueは順序を守って直列実行
- 別リポジトリの実装は、そのリポのパスを指定してサブエージェントに委譲
- **実装完了後**: `npm run type-check`（該当リポで）を確認

### Step 6: 完了サマリー

処理結果を一覧表示:

```
## ingest-references 完了

### トリアージ結果
| 判定 | 件数 |
|------|------|
| HIGH | X件 |
| MEDIUM | X件 |
| DISCARD | X件 |

### 作成/更新したナレッジ
- {ファイルパス}: {追加内容の概要}

### 作成したIssue
- {リポ}#{番号}: {タイトル}

### 実装結果
- {リポ}#{番号}: {完了 / スキップ / エラー}

### まいへの反映
- {反映内容があれば}
```

## エッジケース対応

| ケース | 対応 |
|--------|------|
| 同一URLの重複ファイル | INDEX.mdで既知のURLならDISCARD |
| 大量ファイル（10件+） | サブエージェントを3-4グループに分割して並列処理 |
| ナレッジファイルが肥大化 | 500行超の場合、サブセクションを別ファイルに分割 |
