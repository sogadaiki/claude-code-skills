# Code Review: $ARGUMENTS

## レビュー強度の判定

- `--strict` が含まれる場合 → **厳格モード**(シニアエンジニア視点)
- それ以外 → **通常モード**

---

## 通常モード

対象を以下の5軸でレビュー:

1. **セキュリティ**: XSS, SQLi, 認証漏れ, RLSポリシー
2. **型安全**: any使用、null安全、strict mode準拠
3. **パフォーマンス**: N+1、不要レンダリング、バンドルサイズ
4. **可読性**: 命名規則、複雑度、自己文書化
5. **テスト**: カバレッジ、エッジケース

問題を発見したら修正案を提示。

---

## 厳格モード (--strict)

シニアエンジニアとして、この変更を厳しくレビューする。

### 設計の妥当性
- なぜこの設計を選んだのか？他の選択肢は？
- 将来の拡張性は確保されているか？
- SOLID原則に違反していないか？

### エッジケース
- null/undefined/空配列/空文字列の処理
- 並行処理、レースコンディション
- エラーパスの網羅性

### セキュリティ
- 入力値の検証(バウンダリ検証)
- 認証・認可の漏れ
- データ漏洩リスク

### パフォーマンス
- N+1問題、不要な再レンダリング
- メモリリーク、リソース解放

### テスト
- カバレッジは十分か
- 失敗ケース・境界値のテストはあるか

**問題が見つかった場合:**
「今わかっている全てを踏まえて、エレガントな解決策を実装して」

### 使い方

```
/review src/components/Auth.tsx
/review src/lib/stripe.ts --strict
/review --strict  (直近の変更全体を厳格レビュー)
```
