# Resume Session

**設計原則**: 最小限の情報で最速復帰。Issue一覧やリポ横断スキャンはしない。

## /morning との使い分け

| スキル | 用途 | 実行時間 |
|--------|------|----------|
| `/morning` | 朝一番の全体把握（Issue横断、リポ状況、作業提案） | 重い |
| `/resume` | セッション間の復帰（restart.mdから続きを拾う） | 軽い |

`/finish-session` → `/resume` が標準フロー。
`/morning` は1日1回、朝に実行する。

---

## 手順

### 1. 前回の続きを把握

```bash
git status --short
git log --oneline -3
```

`restart.md` が存在すれば読んで表示する。

### 2. 即座に作業提案

restart.md の「次にやること」の先頭項目をそのまま提案する。

- 「前回の続き: {restart.mdの次にやることの1番目}」
- Issue番号があれば明示

### 3. ナレッジ活性化

ユーザーが作業を選択（または提案を承認）したら:

#### 3.1 設計書の自動読み込み（CRITICAL — スキップ禁止）

> **CRITICAL**: タスクの内容や規模に関係なく、該当する設計書のセクション0は**必ず読む**。
> 「軽微な修正だから不要」という判断は禁止。5行読むだけで致命的ミスを防げる。

タスクのキーワードから該当する設計書を特定し、**セクション0（コンテキスト）を必ず読む**:

```bash
ls ~/.claude/projects/<project-memory-path>/memory/*-architecture.md 2>/dev/null
```

- タスクに含まれるキーワード（クライアント名、プロダクト名、slug）で該当ファイルを特定
- 該当ファイルの**セクション0（コンテキスト）を必ず読む** → 人・関係性・地雷がわかる
- 必要に応じて残りのセクションも読む（費用確認、技術詳細等）
- セクション0があれば、ソースファイル（HTML提案書等）の全文再読み込みは不要
- **鮮度確認**: セクション0を読んだ後、最終更新日が30日以上前なら「この情報は現在も正確ですか？」と1行確認する

#### 3.2 INDEX.mdクラスタの活性化（補助）

設計書だけでは不足する場合のみ:
1. `~/.claude/knowledge/INDEX.md` を読む
2. タスクのキーワードでクラスタを特定（`(memory)` 表記は設計書ファイルを指す）
3. クラスタのファイル群 + `also:` 先を読み込み（1段階まで）

## やらないこと

- 全リポのIssue一覧取得（→ /morningでやる）
- リポ横断のgit status確認（→ /morningでやる）
- 月初ヘルスチェック（→ /morningでやる）
- 作業提案の複数候補提示（→ restart.mdの先頭を提案するだけ）
