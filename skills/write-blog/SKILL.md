# Write Blog: $ARGUMENTS

プロダクトに特化したブログ記事を生成する。
どのプロダクト/ブログでも対応可能。ナレッジファイルから文体・目的・ターゲット・過去記事データを自動で参照する。

## Multi-Pass Design

既存ワークフロー（Step 1-11）を多段階原則にマッピング。

| Pass | 内容 | 対応Step |
|------|------|----------|
| **Pass 1: 素材収集** | テーマ特定、プロダクトナレッジ読込、4並列リサーチ（1次資料/市場/競合/SEO）、スクショ撮影 | Step 1-3.5 |
| **Pass 2: 設計 + 自己批判** | ペルソナ具体化→疑念5つ先回り→記事構成設計。「SEOキーワード自然に入ってるか？」「疑念5つ全部解消する構成か？」を自己批判 | Step 4-6 |
| **Pass 3: 生成** | 初期案執筆（文体/危機感/AI臭さ排除/エビデンスチェック/SEO適用） | Step 7 |
| **Pass 4: 品質ゲート** | 4段並列レビュー（A:信頼性 / B:AI臭検査 / C:Fact-Check / D:SEO/CTA） | Step 8 |
| **Pass 5: 修正** | FAIL項目のみ最小限修正→再チェック。2回FAILで該当セクション再設計 | Step 9 |

### Pass 4 品質ゲート項目（ブログ固有）

| Group | 主要チェック |
|-------|------------|
| A: Credibility | 疑念5つ解消/胡散臭さ/説得力/読了動機/感情的反発 |
| B: AI臭検査 | 禁止表現/体言止め/3点セット/文末反復/構造反復/表乱用/箇条書き乱用/太字過多/見出し連続 |
| C: Fact-Check | 数値正確性/出典完全性/データ鮮度/1次資料優先/薬機法/景表法/著作権/機密情報 |
| D: SEO/CTA | KWタイトル/KW見出し/KW本文冒頭/関連KW/メタディスク/CTA/内部リンク/ターゲット整合 |

## 手順

### 1. テーマの特定

引数がIssue番号の場合: `gh issue view` で詳細取得
引数がテーマ文字列の場合: そのまま使用

### 2. プロダクト情報の読み込み

対象プロダクトのナレッジファイルを参照:

```
~/.claude/knowledge/product-{product}.md
```

**読み込む情報:**
- 事業概要・ターゲット層
- 文体ガイド（存在する場合はナレッジ内のセクションに従う）
- 過去記事一覧・パフォーマンスデータ
- コンテンツ戦略・エビデンスチェックルール
- 競合状況・差別化ポイント

ナレッジに**文体ガイド**が定義されている場合、ステップ7のデフォルト文体ルールよりも**ナレッジの文体ガイドを優先**する。

### 3. リサーチ（サブエージェント並列実行）

以下をサブエージェント(general-purpose)で並列実行:

**3a. 1次資料リサーチ（必須）**
- テーマに関連する**公式サイト・政府統計・業界団体の公式発表**を調査
- 年次更新データ（試験日程、料金、制度変更等）は**必ず公式サイトで最新版を確認**
- 出典URL・アクセス日を記録

**3b. 市場調査**
- テーマに関連する最新トレンド・統計データをWeb検索

**3c. 競合記事分析**
- 同テーマの既存記事の構成・切り口を調査

**3d. SEOキーワード調査**
- 関連キーワード・検索意図の分析

### 3.5. スクリーンショット撮影（参照ページ）

1次資料の公式ページをPlaywrightでスクショ撮影する:

```bash
npx playwright screenshot --viewport-size="1440,900" --full-page "{URL}" {output_path}
```

**保存先**: プロジェクトの公開ディレクトリ
**公開URL**: デプロイ先に応じて設定

スクショは記事内の「出典」表示に使用する。読者が1次資料の実物を視覚的に確認でき、記事の信頼性が向上する。

### 4. 読者ペルソナの具体化

テーマに合わせて想定読者を1人、具体的に定義する。以下を全て決める:

- 年齢・職業・年収
- 現在の悩み・痛み
- 過去に失敗した経験（疑り深さの理由）
- 使う言葉遣い・情報収集の手段

ナレッジにターゲット層が定義されている場合はそれに従う。

### 5. 読者の疑念を5つ先回りする（DOUBT_PREEMPT）

執筆前に、ステップ4のペルソナが今抱いている「疑念」「反論」「そうは言っても」を5つ挙げる。
記事の構成で、この5つを全て解消する論理のステップを組む。解消できていない疑念が1つでもあれば構成をやり直す。

### 6. 記事構成の設計

H2とH3の2階層で十分。H4以下は使わない。見出しの間には最低2〜3段落の本文を入れる。見出し→見出しの連続は禁止。

```
# タイトル（SEO最適化、32文字以内）

リード文（いきなり本題。「この記事では〜を解説します」禁止）
→ 冒頭にパンチライン: 読者の指を止める1行を置く

## 見出し1（問題提起 — 危機感を刻む）
## 見出し2（解決策 — 疑念を1つずつ潰す）
## 見出し3（具体例 — 感情と情景で未来を描く）
## 見出し4（行動喚起 — 読者が今すぐやるべき1手）

→ 末尾にパンチライン: 読了後に行動を起こさせる1行で締める
```

### 7. 初期案の執筆

**ナレッジに文体ガイドがある場合はそちらを優先。** 以下はデフォルトルール。

#### 文体（デフォルト）
- 平文（ですます調 or だ・である調はテーマに合わせて判断。混在禁止）
- 一文は60文字以内を目安。長い文は2つに分ける
- 段落は3〜5文で区切る
- 段落の最初の1文でその段落の結論を言う
- 接続詞で段落をつなげる
- 主語と述語の距離を近く保つ
- スマホでスクロール読みする前提

#### 危機感ドリブン（CRISIS_INJECT）
- 単にやり方を教えるだけの記事は書くな
- 「これを知らないとどうなるか」の恐怖を先に描け
- 読者が「読まずにはいられない」と感じる理由を冒頭で作れ

#### 感情と情景で描写する
- 得られる未来を「利益」ではなく「感情」と「情景」で描け
- 「頑張れば」「誰でも」は耳障りが良いだけ。消せ

#### AI臭さを消す鉄則（最重要）

**絶対禁止パターン:**
- 「いかがでしたか？」系の締め
- 「まとめると」「〜ではないでしょうか」
- 冒頭の「〇〇について解説します」
- 体言止め（動詞で終えろ）
- 「〜することができます」→「〜できる」
- 「まず〜、次に〜、最後に〜」の3点セット構文

**構造ルール:**
- 表（テーブル）は数値データの比較のみ。それ以外は文中に埋め込む
- 箇条書き乱用禁止。3項目以下は文中に書く
- 太字は1セクション1〜2箇所が上限
- 同じ構造の繰り返し禁止
- **同じ文末表現が3回続いたら書き直せ**

**読みやすさの技術:**
- 具体例を入れる
- 数字を使え
- 固有名詞を恐れるな
- 筆者の一人称視点・経験を織り込む
- 読者に話しかけるトーン

#### エビデンスチェック（必須）

記事内の事実・データに対して以下を検証:

1. **1次資料の明示**: 全ての事実・数値に出典を付ける。公式サイト > 報道 > 個人ブログ
2. **年次更新データ**: 日程・料金・制度は最新年度の公式情報を確認
3. **統計データの鮮度**: 3年以上前のデータは「〇〇年時点」と明記
4. **画像引用**: スクショを使用する場合は出典（サイト名+URL）を必ず表記
5. **エビデンス一覧**: 記事末尾に「エビデンス一覧（1次資料）」テーブルを設置

#### SEO
- メインキーワードをタイトル・H2・本文冒頭に自然に配置
- 関連キーワードを本文中に散りばめる
- メタディスクリプション（120文字以内）を末尾に記載

#### 機密情報ガード
以下はナレッジに含まれていても記事に書くな:
- 未公開の売上計画・P/L・単価設計
- 口コミ獲得コスト等の内部KPI
- 資金調達・投資家向け情報
- クライアント固有情報

### 8. 4段並列レビュー（PARALLEL_REVIEW）

初期案を書いたら、そのまま出すな。4つの専門レビューを**並列サブエージェント**で同時実行する。

#### 実行方法
4つのサブエージェント（general-purpose）を同時に起動する。各エージェントには記事Markdown全文、ペルソナ情報（Step 4）、疑念5つ（Step 5）、担当グループのチェック項目を渡す。

#### Review A: Credibility（読者ペルソナ視点の信頼性）

ステップ4のペルソナになりきり、疑念5つ（Step 5）が解消されているかを検証する。

| # | チェック項目 | 基準 | FAIL時の修正指示フォーマット |
|---|-------------|------|---------------------------|
| A1 | 疑念解消 | Step 5の疑念5つが全て記事内で明確に解消されているか。1つでも未解消ならFAIL | 「疑念'{疑念内容}'が未解消。{該当セクション}に以下を追加: {具体的な解消文}」 |
| A2 | 胡散臭さ | ペルソナが「胡散臭い」と感じる箇所がないか。根拠なき断言、大げさな表現 | 「{該当箇所}が胡散臭い。修正: {具体的な代替文}」 |
| A3 | 説得力 | 主張に対する根拠が十分か。「なぜそう言えるのか」が示されているか | 「{該当主張}に根拠不足。{エビデンス or 具体例}を追加」 |
| A4 | 読了動機 | 冒頭で「読む気が失せる」箇所がないか。離脱ポイントを特定 | 「{該当箇所}で離脱リスク。修正: {改善案}」 |
| A5 | 感情的反発 | 「上から目線」「煽りすぎ」と感じる箇所がないか | 「{該当箇所}がペルソナに反発される。トーン修正: {代替表現}」 |

**判定**: 全項目PASSなら `{"group": "A", "status": "PASS"}`。1つでもFAILなら `{"group": "A", "status": "FAIL", "failures": [...]}`

#### Review B: AI臭検査

記事がAI生成っぽく見えないかを検証する。

| # | チェック項目 | 基準 | FAIL時の修正指示フォーマット |
|---|-------------|------|---------------------------|
| B1 | 禁止表現 | 「いかがでしたか」「まとめると」「〜ではないでしょうか」「〜について解説します」「〜することができます」が含まれていないか | 「{該当表現}を削除/置換: {代替表現}」 |
| B2 | 体言止め | 文末が体言止めになっていないか（動詞で終えるルール） | 「{該当箇所}を動詞終止に修正: {修正案}」 |
| B3 | 3点セット構文 | 「まず〜、次に〜、最後に〜」パターンがないか | 「{該当箇所}の構文を崩す: {修正案}」 |
| B4 | 文末反復 | 同じ文末表現が3回以上連続していないか | 「{該当箇所}の文末を変化させる: {修正案}」 |
| B5 | 構造反復 | 同じ構造（見出し→箇条書き→まとめ等）が繰り返されていないか | 「{セクション}の構造を変える: {修正案}」 |
| B6 | 論理の飛躍 | 根拠なく結論に飛んでいる箇所がないか | 「{該当箇所}に論理の橋を追加: {修正案}」 |
| B7 | 表の乱用 | 数値比較以外で表を使っていないか | 「{該当テーブル}を本文に埋め込む: {修正案}」 |
| B8 | 箇条書き乱用 | 3項目以下の箇条書きがないか | 「{該当リスト}を文中に統合: {修正案}」 |
| B9 | 太字過多 | 1セクションに太字が3箇所以上ないか | 「{セクション}の太字を{N}箇所に削減」 |
| B10 | 見出し連続 | 見出し直後にまた見出しが来ていないか（間に本文2-3段落必須） | 「{該当見出し}の間に本文を挿入: {具体案}」 |

**判定**: 全項目PASSなら `{"group": "B", "status": "PASS"}`。1つでもFAILなら `{"group": "B", "status": "FAIL", "failures": [...]}`

#### Review C: Fact-Check（事実検証専門）

記事内の全ての事実・数値・法的表現を専門的に検証する。**このレビューのみ、WebSearchツールを使って1次資料を再確認すること。**

| # | チェック項目 | 基準 | FAIL時の修正指示フォーマット |
|---|-------------|------|---------------------------|
| C1 | 数値の正確性 | 記事内の全ての数値（金額・日程・統計・割合）が1次資料と一致するか。WebSearchで公式サイトを再確認 | 「{該当数値}が不正確。正しくは{正確な値}（出典: {URL}）」 |
| C2 | 出典の完全性 | エビデンス一覧に全ての事実・数値の出典が記載されているか | 「{該当事実}の出典が未記載。追加: {出典情報}」 |
| C3 | データ鮮度 | 3年以上前のデータに「〇〇年時点」の注記があるか | 「{該当データ}に年次注記を追加: '{年}年時点'」 |
| C4 | 1次資料優先 | 出典が公式サイト > 報道 > 個人ブログの優先順位に従っているか | 「{該当出典}を1次資料に差し替え: {公式サイトURL}」 |
| C5 | 薬機法 | 医薬品・健康食品・化粧品に関する効能効果の断定表現がないか | 「{該当表現}が薬機法リスク。修正: {代替表現}」 |
| C6 | 景表法 | 「No.1」「最安」「確実に」等の優良誤認・有利誤認表現がないか | 「{該当表現}が景表法リスク。修正: {代替表現}」 |
| C7 | 著作権 | 他サイトからの引用が引用の要件（主従関係・出所明示・必然性）を満たしているか | 「{該当引用}の引用要件不備。修正: {具体的な対応}」 |
| C8 | 機密情報 | 未公開売上・P/L・単価設計・内部KPI・クライアント固有情報が含まれていないか | 「{該当情報}は機密。削除すること」 |

**判定**: 全項目PASSなら `{"group": "C", "status": "PASS"}`。1つでもFAILなら `{"group": "C", "status": "FAIL", "failures": [...]}`

#### Review D: SEO/CTA（マーケティング視点）

マーケティング視点でSEO・CTA・コンバージョン導線を検証する。マーケティング専門のサブエージェントに委譲推奨。

| # | チェック項目 | 基準 | FAIL時の修正指示フォーマット |
|---|-------------|------|---------------------------|
| D1 | KWタイトル配置 | メインキーワードがタイトルに含まれているか（前半推奨） | 「タイトルにKW'{キーワード}'を配置: {修正案}」 |
| D2 | KW見出し配置 | メインキーワードがH2に自然に含まれているか | 「{H2}にKWを自然に統合: {修正案}」 |
| D3 | KW本文冒頭 | メインキーワードが本文冒頭200文字以内に出現するか | 「冒頭にKWを自然に配置: {修正案}」 |
| D4 | 関連KW散布 | 関連キーワードが本文中に自然に散りばめられているか | 「{関連KW}を{セクション}に追加: {修正案}」 |
| D5 | メタディスクリプション | 120文字以内か、KWを含むか、クリック誘引力があるか | 「メタディスクリプションを修正: {修正案}」 |
| D6 | CTA適切性 | CTAがターゲット読者に響くか。押し売り感がないか。配置位置は適切か | 「{位置}のCTAを修正: {具体案}」 |
| D7 | 内部リンク | 関連する過去記事・プロダクトページへのリンクが配置されているか | 「{セクション}に内部リンクを追加: {リンク先とアンカーテキスト}」 |
| D8 | 読者ターゲット整合 | 記事全体のトーンとメッセージがターゲット読者に合っているか | 「{該当箇所}のトーンがターゲットに不適合。修正: {方針}」 |

**判定**: 全項目PASSなら `{"group": "D", "status": "PASS"}`。1つでもFAILなら `{"group": "D", "status": "FAIL", "failures": [...]}`

#### 結果集約

4グループの結果をマージし、レビュー結果を記録する:
```json
{
  "groups": [
    {"group": "A", "status": "PASS|FAIL", "failures": [{"id": "A1", "detail": "..."}]},
    {"group": "B", "status": "PASS|FAIL", "failures": [{"id": "B3", "detail": "..."}]},
    {"group": "C", "status": "PASS|FAIL", "failures": [{"id": "C1", "detail": "..."}]},
    {"group": "D", "status": "PASS|FAIL", "failures": [{"id": "D6", "detail": "..."}]}
  ],
  "overall": "PASS|FAIL",
  "timestamp": "ISO8601"
}
```

### 9. 最小限修正

Step 8でFAILしたグループの修正指示**のみ**を反映した最終版を生成する。
PASSしたグループの該当箇所には触らない（最小限修正の原則）。
修正後、FAILしたグループの項目のみ再チェックし、全PASSを確認してから最終版とする。

### 10. 成果物の保存

#### 10a. Markdown原稿

```
blog/{product}/YYYY-MM-{slug}.md
```

ファイル冒頭にfrontmatterを付与:
```yaml
---
title: "記事タイトル"
date: YYYY-MM-DD
product: {product}
author: "{著者名}"
tags: [tag1, tag2, tag3]
description: "メタディスクリプション"
status: draft
---
```

#### 10b. HTML（WordPress投稿用）

Markdown原稿からHTML版を生成:

```
blog/{product}/YYYY-MM-{slug}.html
```

**HTML要件:**
- 画像は公開URLを `<img>` で参照
- 出典はキャプション付きリンク（`<p class="img-caption">出典: <a href="...">サイト名</a></p>`）
- テーブル・リスト・CTAはHTML構造で記述
- メタ情報（KW・カテゴリ・内部リンク先）は冒頭に配置
- ブラウザで開いてWordPressにそのままコピペできる品質

#### 10c. スクショ画像のデプロイ

デプロイ先に応じてコマンドを実行。

#### 10d. ダッシュボード反映

ダッシュボードの該当クライアントに deliverable を追加。

### 11. 完了

以下を報告:
- 記事ファイルパス（.md + .html）
- 文字数
- SEOキーワード
- ペルソナ（誰に向けて書いたか）
- 解消した疑念5つ
- エビデンス一覧（1次資料URL）
- スクショ画像枚数
- ダッシュボード更新状況
