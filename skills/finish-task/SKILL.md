# Finish Task

タスクを完了し、mainにコミット・プッシュする。ブランチ・PR不要の通常ワークフロー用。

> **並列開発中（ブランチ運用時）**: ブランチ上にいる場合は従来通りPR作成→マージ→mainに戻るフローを実行する

## 手順

### 1. 現在の状態を確認

```bash
git branch --show-current
git status
git diff --stat
```

- mainブランチにいるか確認
- ブランチ上にいる場合 → PR作成フローに切り替える（後述）

### 2. 変更内容の確認とコミット

未コミットの変更がある場合:

1. `git status` で変更ファイルを確認
2. 変更内容を分析して適切なコミットメッセージを作成
3. 関連ファイルをステージング（`git add -A` ではなく個別指定を優先）
4. コミット（Issue番号がある場合は含める）

```bash
git add {files}
git commit -m "{type}: {簡潔な説明}"
```

**コミットメッセージのフォーマット**:
```
{type}: {簡潔な説明}

{詳細（必要な場合）}

Co-Authored-By: Claude <noreply@anthropic.com>
```

変更がない場合はコミットをスキップ。

### 3. プッシュ

```bash
git push origin main
```

### 4. 関連Issueのクローズ

引数でIssue番号が指定された場合、またはセッション中の会話からIssue番号が特定できる場合:

```bash
gh issue close {issue-number} --comment "完了。{簡潔な完了サマリー}"
```

Issue番号が不明な場合はスキップ（ユーザーに聞かない）。

### 5. restart.md を更新

#### コンテキスト判定

```bash
git remote get-url origin 2>/dev/null
```

- メインopsリポを含む → 全リポジトリのオープンIssueを取得（司令塔モード）
- それ以外 → `gh issue list --state open --limit 10`（このリポのみ）

#### 司令塔モードのIssue取得:
```bash
# 自分のリポジトリに合わせて変更
gh issue list --repo <your-org>/<main-repo> --state open --limit 10
gh issue list --repo <your-org>/<project-a> --state open --limit 10
```

#### プロジェクトモードのIssue取得:
```bash
gh issue list --state open --limit 10
```

プロジェクトルートの `restart.md` に新しいセッションを先頭に追加する。
既存の `## Session:` ブロックは最大2つまで保持。

```markdown
# Restart Notes

## Session: {今日の日付}
Branch: main

### 完了したこと
{このセッションで完了したタスク・Issue}

### 次にやること
{残りのオープンIssueを優先順位付きで記載}
{各Issueに1行で理由を添える}

### 注意点
{引き継ぎで知っておくべきこと。なければ省略}
```

### 6. 作業ログを追記

`<ops-repo-path>/logs/{YYYY-MM-DD}.md` に作業内容を追記する。
**どのリポジトリで実行しても、常にメインopsリポの `logs/` に書き込む。**

ファイルが存在しない場合は新規作成（見出しを先頭に）。存在する場合は末尾に追記。

```markdown
# {YYYY-MM-DD} 作業ログ          ← ファイル新規作成時のみ

---

## {HH:MM} {リポジトリ名}
**ステータス**: 完了
**Issue**: #{番号}（あれば）

### やったこと
- {箇条書き、具体的に}

### 成果物
- {作成・変更したもの: ページ、API、テーブル等}

### タグ
{該当するものだけ: #development #design #marketing #content #bugfix #refactor #infra #client}
```

**ルール:**
- セッションの会話から自動で判断する。ユーザーに聞かない
- 「やったこと」は第三者が読んでも分かる粒度で書く
- 「成果物」は具体的なアウトプットを列挙（finish-taskはタスク完了なので成果物がある）
- タグは1-3個。後からgrepで検索できるようにする
- 時刻はログ書き込み時点のローカル時刻（`date +%H:%M`）

### 7. ナレッジ保存チェック

セッション中の会話を振り返り、以下に該当するものがあれば該当するメモリ/ナレッジファイルに追記する。
**ユーザーに聞かずに自己判断で実行する。**

#### 保存すべきもの（1つでも該当すれば保存）

| 種類 | 例 | 保存先 |
|------|-----|--------|
| **新しい事実・設定値** | API仕様、アカウント情報、外部サービスの挙動 | memory/ or knowledge/ の該当ファイル |
| **成果物の詳細** | 作成した原稿の内容要約、設計判断の根拠 | memory/ or knowledge/ の該当ファイル |
| **失敗・教訓** | エラーの原因、やってはいけないこと | 該当ファイル + `/learn-mistake` 相当の記録 |
| **外部サービスの制約** | 規約の解釈、審査基準、料金体系 | knowledge/ の該当ファイル |
| **クライアント固有情報** | 確認事項、決定事項、ステータス変更 | memory/ の該当ファイル |

#### 保存しないもの

- 一時的な作業状態（restart.mdで十分）
- コードの変更内容そのもの（gitで追える）
- 推測・未確定の情報（確定したら保存）

#### 実行手順

1. 上記の表に該当する情報がセッション中にあるか振り返る
2. 該当あり → 適切なファイルに追記（既存セクションの更新 or 新規セクション追加）
3. 該当なし → スキップ（「保存すべきナレッジなし」と報告のみ）
4. 保存した場合 → 完了報告に「ナレッジ保存: {ファイル名} に {内容の1行要約}」を追加

**判断基準**: 「次回このテーマに取り組む時、この情報がなかったら困るか？」 → YESなら保存。

### 8. 完了報告

以下を表示:
- コミットハッシュとメッセージ
- クローズしたIssue（あれば）
- 残りのオープンIssue一覧（優先順位付き）
- 作業ログに追記した旨
- ナレッジ保存した内容（あれば）
- 「次回 `/morning` で再開できます」

## ブランチ上にいる場合（並列開発時）

mainではなくブランチ上にいる場合は以下のフローに切り替える:

1. 変更をコミット
2. `git push -u origin {branch-name}`
3. `gh pr create` でPR作成（`Closes #{issue}` を含める）
4. PRのURLを表示
5. `git checkout main && git pull origin main`
6. restart.md を更新
