# 事業計画書レビュー

レビュー対象: $ARGUMENTS

## 参照ナレッジ
- `~/.claude/knowledge/agent-teams.md` — チーム運用パターン
- `~/.claude/knowledge/team-principles.md` — 共通行動原則

---

## Phase 1: 個別分析（並列）

以下のサブエージェントチームを**並列**で起動し、各自が独立にレビューする:

1. **mei-business-strategist**: 事業戦略の妥当性、市場分析、競合優位性
2. **rina-finance-advisor**: 財務計画の整合性、収益モデル、キャッシュフロー
3. **kenji-marketing-strategist**: マーケティング戦略の実現可能性
4. **hiroshi-legal-advisor**: 法的リスク、コンプライアンス要件

### 各エージェントへの指示

各エージェントに以下の観点を割り当てる:

**mei**: ストーリーライン（投資家視点）+ 市場分析
- 共感を得られるか？数字で裏付けられているか？
- 「なぜこの会社がやるべきか」が明確か？
- TAM/SAM/SOMは妥当か？出典が明記されているか？
- 競合分析は十分か？

**rina**: 財務計画
- 前提条件は明記されているか？
- 3シナリオ（保守的/基本/楽観的）があるか？
- キャッシュフローの時期は整合しているか？

**kenji**: 実行可能性 + マーケティング
- チーム体制は十分か？
- タイムラインは現実的か？マイルストーンは測定可能か？
- 顧客獲得コストは現実的か？

**hiroshi**: リスク開示
- 主要リスクが全て列挙されているか？
- 対策は具体的か？投資家にとって誠実な開示になっているか？

### Phase 1 出力フォーマット（各エージェント）

各エージェントは以下のJSON構造で報告すること:

```json
{
  "agent": "mei|rina|kenji|hiroshi",
  "evaluation": "A|B|C|D|F",
  "strengths": ["..."],
  "critical_issues": ["..."],
  "questions_for_peers": [
    {
      "to": "rina|kenji|mei|hiroshi",
      "question": "具体的な質問内容"
    }
  ]
}
```

`questions_for_peers` は Phase 2 で使用する。各エージェントは最低1つ、他のエージェントへの質問を含めること。

---

## Phase 2: ピア議論（SendMessage、最大2往復）

Phase 1 の報告が全て揃った後、エージェント間で相互質問・検証を行う。

### 議論のルーティング

以下の3つの議論ラインを**並列**で実行する:

1. **mei → rina**: 戦略 x 財務の整合性
   - 例: 「TAMが妥当なら保守シナリオで収支合うか？」
   - 例: 「市場参入タイミングとキャッシュバーン期間の整合性は？」

2. **rina → kenji**: 財務 x マーケティングの整合性
   - 例: 「この顧客獲得コストは実現可能か？」
   - 例: 「LTV/CAC比率は業界水準と比較してどうか？」

3. **kenji → mei**: マーケティング x 戦略の整合性
   - 例: 「市場要求と戦略の整合性は？」
   - 例: 「ターゲット顧客のペインポイントとプロダクトの訴求は一致しているか？」

hiroshi は法務観点から、上記3ラインの議論結果にコメントを追加する（法的リスクが発見された場合のみ）。

### 議論制約
- **最大2往復**（質問→回答→再質問→再回答）。それ以上は収穫逓減
- 各メッセージは **200文字以内** に簡潔にまとめる
- 合意に達した点、未解決の論点を明確に分離する

### Phase 2 出力フォーマット（議論記録）

```json
{
  "discussions": [
    {
      "line": "mei-rina|rina-kenji|kenji-mei",
      "exchanges": [
        {"from": "mei", "to": "rina", "message": "..."},
        {"from": "rina", "to": "mei", "message": "..."}
      ],
      "consensus": "合意内容（あれば）",
      "unresolved": "未解決事項（あれば）"
    }
  ],
  "new_insights": ["議論から生まれた新しい洞察"]
}
```

## Phase 3: 統合評価（meiがリーダー）

mei-business-strategist がリーダーとして、以下の全入力を統合する:

**入力:**
- Phase 1: 全エージェントの個別分析レポート
- Phase 2: 議論記録（合意事項 + 未解決事項 + 新しい洞察）

**統合作業:**
1. 全評価を総合して統一評価（A/B/C/D/F）を決定
2. 評価の根拠を議論内容から引用
3. 議論から生まれた新しい洞察を明記（Phase 1 では出なかった結論）

### 最終出力フォーマット

```
### 総合評価: A/B/C/D/F

### 評価根拠
（なぜこの評価になったか。議論の合意事項を引用）

### 強み（Top 3）
1. ...
2. ...
3. ...

### 改善必須事項（Critical Issues）
- ...

### 改善推奨事項（Nice to Have）
- ...

### 議論から生まれた洞察
（個別レビューでは出なかった、ピア議論から生まれた結論を列挙）
- ...

### 未解決の論点
（エージェント間で合意に至らなかった論点。ユーザーの判断が必要）
- ...

### 次のアクション（具体的なTo-Do）
1. ...
2. ...
3. ...
```

## レビュー結果のナレッジ保存（自動実行）

レビュー結果をユーザーに提示した後、以下をナレッジベースに自動保存する:

保存先: 対象に応じて `business.md` または `decisions.md`（`~/.claude/knowledge/` 配下）

保存内容:
```markdown
## {レビュー対象名} レビュー結果 ({今日の日付})

**チーム構成**: {参加したエージェント名}
**総合評価**: {A〜F}
**Critical Issues**: {改善必須事項の箇条書き}
**チーム議論から生まれた洞察**: {個別レビューでは出なかった、議論から生まれた結論}
**未解決の論点**: {合意に至らなかった事項}
**次のアクション**: {具体的なTo-Do}
```

**保存ルール:**
- ユーザーに「保存しますか？」と聞かない。自動で保存する
- 既存のナレッジに追記する形で保存する（上書きしない）
- 保存完了後「レビュー結果をナレッジベースに保存しました」と1行だけ報告する
